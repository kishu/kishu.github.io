이 글은 npm 패키지를 여러 플랫폼에 사용하는 방법을 설명한다. 시작하기 전 자바스크립트의 모듈 형식을 간단히 알아보자.

역자주: package.json에서 소스 진입점을 가리키는 속성은 main, browser, module, es2015 등이 있다. 위 표는 각 속성을 사용하기 위해 필요한 ES 버전과 모듈 형식을 알려준다. 또한 주요 번들링 도구에서 해당 속성을 사용할 수 있는지 알려준다. (✔)는 플러그인을 통해 사용 가능하다는 의미다.

2. 배경지식: 자바스크립트 모듈 형식

현재 일반적인 자바스크립트 모듈 형식은 다음과 같다.

  • AMD(Asynchronous Module Definition): 브라우저용 비동기 모듈 형식
  • CJS(Common JavaScript): Node.js 같은 서버용으로 설계한 동기식 모듈 형식. Node.js와 npm의 인기로 인해 CJS는 브라우저에서 가장 널리 사용하는 형식이다. 하지만 비동기 코드로 컴파일해야 하며, 이를 위해 webpack이나 browserify 등의 도구를 사용한다.
  • ESM(ECMAScript modules): ES6는 모듈을 내장했다. ESM은 동기식과 비동기식에서 모두 사용 가능하게 설계했다. 최근 브라우저에서도 이를 지원하기 시작했다. Node.js도 2018년 초(빠르면 그 전이라도)에 지원할 예정이다.

3. 이슈

npm 패키지는 단일 플랫폼에서만 사용할 수 있게 제공된다. 다음 코드처럼 package.json의 engines 속성을 사용하면 대상 플랫폼을 지정할 수 있다.

하지만 이 속성은 패키지가 여러 플랫폼을 지원해야 하며 각 플랫폼 마다 진입점이 달라야 하는 상황에 전혀 도움을 줄 수 없다. 다음과 같은 상황을 살펴보자.

  1. 브라우저: Babel이 트랜스파일 할 수 있도록 네이티브 버전 (예 : CJS를 통한 ES5)과 ‘최첨단’버전 (예 : ES 모듈을 통한 최신 ECMAScript 버전)을 모두 제공하는 상황.
  2. Node.js: 서로 다른 몇개의 Node.js 버전에 동일한 모듈을 제공해야 하는 상황.
  3. 브라우저: 어떠한 라이브러리를 오류 없이 오래 사용해야 하는 상황 - 타겟 플랫폼이 새 기능을 지원하지 않는 한 오래 사용해야 함. babel-preset-env같은 편리함이 필요한 상황.

이런 상황에는 다음과 같은 두가지 측면을 고려해야 한다.

  • 트랜스파일 할 코드와 네이티브 코드는 분리되어 있다.
  • 네이티브 코드는 서로 다른 플랫폼에서 실행되어야 한다.

다음은 첫번째 상황에 대한 해결책이다.

4. 해결책

다음에 나오는 package.json의 속성들은 동일한 코드의 대체 버전을 가리키기 위해 사용할 수 있다. “native code”로 표시한 것은 타겟 플랫폼에서 지원하는 언어의 기능을 의미한다.

4.1 main: native feature, CJS

main은 패키지 내부의 모듈 코드를 가리키는 표준 매커니즘이다. 기본은 index.js이며 원하는 파일로 바꿀 수 있다. 어떤 환경이든 지원한다.

4.2 module: native feature, ESM

이 속성은 ESM 형식을 지원하며 트리쉐이킹이 가능한 Rollup 같은 도구를 지원한다. 그 외로는 언어의 네이티브 기능으로 지원한다. 즉 module은 또 다른 형태의 모듈에 대한 main이다.

4.3 es2015: ES6, ESM

Angular4는 패키지를 세가지 형식으로 제공한다.

  • UMD: main 프로퍼티를 통해
  • ES5/ESM: module 프로퍼티를 통해
